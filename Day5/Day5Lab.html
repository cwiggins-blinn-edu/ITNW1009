Lab 3, Part 1: Launching & Monitoring Your First EC2 Instance

Launch a protected web server, then explore the tools AWS gives you to monitor it.

Lab Overview

Amazon Elastic Compute Cloud (EC2) is a web service that provides resizable compute capacity in the cloud, letting you obtain and configure virtual servers in minutes and pay only for what you use. In this lab you'll launch a web server with termination and stop protection enabled, monitor it, open it up to HTTP traffic, resize it, and explore account-level EC2 limits. Estimated time: 35 minutes.

Task 1: Launch Your Amazon EC2 Instance

In the AWS Management Console, choose Services → Compute → EC2. Confirm the region dropdown reads N. Virginia (us-east-1), then choose Launch instance → Launch instance.

Name & Image
  • Name: Web Server
  • AMI: keep default Amazon Linux → Amazon Linux 2023
  • Instance type: keep default t2.micro (1 vCPU, 1 GiB RAM)
  • Key pair: choose vockey (not actually used to log in during this lab)
Network Settings
  • Click Edit next to Network settings
  • VPC: select Lab VPC; keep subnet PublicSubnet1
  • Under Firewall, choose Create security group: name it Web Server security group, description "Security group for my web server"
  • Remove the one default inbound rule that appears

Keep the default Configure storage settings (an 8 GiB root EBS volume). Expand Advanced details and set Termination protection to Enable — this prevents the instance from being accidentally deleted.

User Data Bootstrap Script

Scroll to the bottom of Advanced details and paste this into the User data box — it installs and starts Apache, then creates a simple test page:

#!/bin/bash
dnf install -y httpd
systemctl enable httpd
systemctl start httpd
echo '<html><h1>Hello From Your Web Server!</h1></html>' > /var/www/html/index.html

Scroll to the bottom of the Summary panel and choose Launch instance, then View all instances. Wait for Instance State: Running and Status Checks: 2/2 checks passed.

Task 2: Monitor Your Instance

  1. Choose the Status checks tab — confirm both System reachability and Instance reachability have passed.
  2. Choose the Monitoring tab to view Amazon CloudWatch metrics for the instance (basic 5-minute monitoring is enabled by default).
  3. In the Actions menu, choose Monitor and troubleshoot → Get system log to view console output — a valuable tool for diagnosing kernel or service startup problems. Note that the httpd package install shows up here from your user data script. Choose Cancel when done.
  4. With Web Server still selected, choose Monitor and troubleshoot → Get instance screenshot to see what a display attached to the instance would show. Choose Cancel when done.

Lab 3, Part 2: Opening HTTP Access, Resizing & Stop Protection

Fix the firewall so your web page loads, then scale the instance up and test its safety switches.

Task 3: Update Your Security Group & Access the Web Server

On the Details tab, copy the instance's Public IPv4 address. Open a new browser tab, paste it, and press Enter — the page won't load, because the security group doesn't yet allow inbound traffic on port 80.

  1. Back in the EC2 console, choose Security Groups in the left pane and select Web Server security group.
  2. Open the Inbound rules tab (currently empty) and choose Edit inbound rules → Add rule.
  3. Set Type: HTTP, Source: Anywhere-IPv4, then choose Save rules.
  4. Return to the web server browser tab and refresh — you should now see "Hello From Your Web Server!"

Task 4: Resize the Instance Type & EBS Volume

An instance must be stopped before you can change its type. Select Web Server, choose Instance state → Stop instance, confirm, and wait for state Stopped (there's no runtime charge while stopped, though EBS storage charges continue).

  1. With Web Server selected, go to Actions → Instance settings → Change instance type, set Instance Type to t2.small (double the memory of t2.micro), and choose Apply.
  2. Go to Actions → Instance settings → Change stop protection, select Enable, and Save.
  3. Open the Storage tab, select the Volume ID checkbox, then Actions → Modify volume. Change the size from 8 to 10 GiB and choose Modify twice to confirm.
  4. Back in Instances, select Web Server and choose Instance state → Start instance.

Task 5: Explore EC2 Limits

  1. Search for and open Service Quotas in the console.
  2. Choose AWS services, search for ec2, and select Amazon Elastic Compute Cloud (Amazon EC2).
  3. In Find quotas, search for running on-demand (without selecting anything) and observe the filtered list — each AWS account has default, per-region limits on the number of running instances of each type. Account owners can request increases to many of these.

Task 6: Test Stop Protection & Stop the Instance

  1. Return to Instances, select Web Server, and choose Instance state → Stop instance → Stop.
  2. You'll see a failure message about the disableApiStop attribute — this confirms the stop protection you enabled earlier is working.
  3. To actually stop it, go to Actions → Instance settings → Change stop protection, uncheck Enable, and choose Save.
  4. Select Web Server again, choose Instance state → Stop instance → Stop — this time it succeeds.
Lab 3 Complete

Submit your progress using the Submit button at the top of your lab instructions, then move on to Lab 6 in the next tab.

Lab 6, Part 1: Building an AMI, Target Group & Load Balancer

Package your web server into a reusable image, then front it with an Application Load Balancer.

Lab Overview

Elastic Load Balancing automatically distributes incoming traffic across multiple EC2 instances for fault tolerance. Auto Scaling keeps your desired number of instances running, adding capacity during demand spikes and removing it during lulls. In this lab you'll create an AMI, a load balancer, a launch template, and an Auto Scaling group — then trigger and observe a real scaling event. Estimated time: 30 minutes.

Account Safety Limit

Any attempt to run 20 or more concurrent instances (regardless of size) will result in immediate deactivation of the AWS account and deletion of all resources in it. The Auto Scaling group in this lab is capped well below that limit.

Task 1: Create an AMI from Web Server 1

  1. Open the EC2 console, choose Instances, and confirm Web Server 1 shows 2/2 status checks passed.
  2. Select it, then choose Actions → Image and templates → Create image.
  3. Set Image name: WebServerAMI, Image description: "Lab AMI for Web Server", then choose Create image. You'll use this AMI to launch the Auto Scaling group's instances later.

Task 2: Create a Target Group & Application Load Balancer

Create the Target Group

Target groups define where the load balancer sends traffic. In the left pane, choose Target Groups → Create target group:

  • Target type: Instances
  • Target group name: LabGroup
  • VPC: Lab VPC
  • Choose Next, skip registering targets (none exist yet), then Create target group
Create the Application Load Balancer

In the left pane, choose Load Balancers → Create load balancer. Under Application Load Balancer (layer 7, request-level routing), choose Create:

  • Load balancer name: LabELB
  • Network mapping VPC: Lab VPC
  • Availability Zone 1: Public Subnet 1; Availability Zone 2: Public Subnet 2
  • Security groups: select Web Security Group, then remove the default security group so only Web Security Group remains
  • Listener HTTP:80 → Default action: forward to LabGroup
  • Choose Create load balancer, then View load balancer — it will show provisioning; no need to wait before moving on

Lab 6, Part 2: Launch Template, Auto Scaling & Live Load Testing

Wire up Auto Scaling, confirm the load balancer is healthy, then trigger a real scale-out event.

Task 3: Create a Launch Template & Auto Scaling Group

Launch Template

In the left pane, choose Launch Templates → Create launch template:

  • Name: LabConfig; enable Auto Scaling guidance
  • AMI: choose My AMIs → Web Server AMI
  • Instance type: t2.micro; Key pair: vockey
  • Firewall: select existing → Web Security Group
  • Expand Advanced details, enable Detailed CloudWatch monitoring so Auto Scaling can react quickly to changing load
  • Choose Create launch template
Auto Scaling Group

From the Success dialog, select LabConfig, then Actions → Create Auto Scaling group:

  • Name: Lab Auto Scaling Group
  • VPC: Lab VPC; Subnets: Private Subnet 1 and Private Subnet 2
  • Attach to existing load balancer target group: LabGroup; enable group metrics collection in CloudWatch
  • Group size — Desired: 2, Minimum: 2, Maximum: 6
  • Scaling policy: Target tracking, name LabScalingPolicy, metric Average CPU Utilization, target value 60
  • Tags: Key Name, Value Lab Instance
  • Review and choose Create Auto Scaling group — it will grow from 0 toward the desired count of 2

Task 4: Verify Load Balancing Is Working

  1. Under Instances, confirm two new instances named Lab Instance have launched (refresh after ~30 seconds if not yet visible).
  2. Go to Target Groups → LabGroup → Targets tab, and wait until both instances show status healthy.
  3. Go to Load Balancers → LabELB, copy its DNS name (omit "(A Record)"), open a new tab, paste it, and press Enter — the web application should load, confirming the load balancer is routing traffic correctly.

Task 5: Trigger & Observe Auto Scaling

  1. Open CloudWatch → All alarms. Two alarms were auto-created by the Auto Scaling group to keep average CPU near 60% within the 2–6 instance range. If they aren't visible within 60 seconds, edit LabScalingPolicy (under the Auto Scaling group's Automatic Scaling tab) and set Target Value to 50 to help trigger them.
  2. Choose the alarm with AlarmHigh in its name — it should read OK, meaning CPU is currently low.
  3. Return to the web application browser tab and choose Load Test next to the AWS logo. The page auto-refreshes and drives CPU load across the Auto Scaling group's instances. Leave this tab open.
  4. Back in CloudWatch (refresh every ~60 seconds), within about 5 minutes AlarmLow should read OK and AlarmHigh should switch to In alarm once CPU crosses 60% for more than 3 minutes.
  5. Return to EC2 Instances — you should now see more than two Lab Instance entries, added automatically in response to the alarm.

Task 6: Terminate Web Server 1

Web Server 1 was only needed to build the AMI — it's no longer required. Select only Web Server 1, choose Instance state → Terminate Instance, and confirm Terminate.

Lab 6 Complete

Choose Submit at the top of your lab instructions to record your score, then End Lab and confirm to tear down the sandbox and its resources.

Day 5 Summary: Key Takeaways

•

Protection Settings Prevent Accidents

Termination protection and stop protection are independent safeguards that block accidental deletion or shutdown — both must be disabled deliberately before the action succeeds.

•

Security Groups Gate Everything

No inbound rule means no traffic — even a fully running, healthy web server is unreachable until its security group explicitly allows the port.

•

AMIs Make Instances Reproducible

Capturing a configured instance as an AMI lets Auto Scaling launch identical, pre-configured servers on demand.

•

Target Tracking Keeps Load Balanced

A CloudWatch-driven target tracking policy automatically adds or removes instances to hold average CPU near a defined target, scaling within a safe min/max range.
