Task 2: Create the DB Security Group (DBSG)

To enforce the Principle of Least Privilege, your database must never be directly exposed to the internet. You'll create a dedicated security group that accepts MySQL traffic on Port 3306 only from the web server's security group.

Traffic Flow Architecture

Traffic flows in one direction — inbound to the database — and only from trusted sources:

Internet → WebSG → Web Server → DBSG (3306) → RDS Instance

By setting the inbound rule source to the Web-Security-Group ID (not an IP range), AWS automatically grants access to any instance belonging to that group — a dynamic, scalable, secure design pattern.

  1. In the top search bar, type VPC and select VPC. In the left navigation under Security, choose Security groups. Click Create security group.
  2. Set Security group name: DBSG; Description: DB Security Group. For VPC, select Lab VPC — do not leave it on the default VPC.
  3. Scroll to Inbound rules → click Add rule. Set Type to MySQL/Aurora (3306). Set Source type to Custom, then search for and select Web-Security-Group.
  4. Scroll to the bottom and click Create security group. Verify DBSG appears in the Security Groups list.

2. Creating a DB Subnet Group

Give RDS the placement flexibility it needs to run a true Multi-AZ deployment.

Why This Matters

Amazon RDS uses a DB Subnet Group to determine which subnets and Availability Zones it may use when provisioning primary and standby database instances. By spanning two private subnets across two Availability Zones, you enable true Multi-AZ High Availability.

Most Common Error

While You Wait: What Is Multi-AZ Doing?

Primary Instance

AWS provisions the main MySQL instance in your first selected Availability Zone. All reads and writes flow through it during normal operation.

Synchronous Standby Replica

A second, identical instance is provisioned in a different AZ. Every committed transaction is synchronously replicated before the write is acknowledged.

Automatic Failover

If the primary fails, RDS automatically promotes the standby — typically within 60–120 seconds — with no manual intervention required.

4. Connecting the Web Application & Validating the Lab

Task 5: Connect the Web Application to RDS

  1. Copy the RDS Endpoint. In the RDS Console, click lab-db. Under the Connectivity & security tab, find the Endpoint string (e.g., lab-db.c1234567890.us-east-1.rds.amazonaws.com). Copy it.
  2. Get the Web Server's Public IP. Navigate to EC2 → Instances. Select the Web Server checkbox and copy the Public IPv4 address from the Details pane.
  3. Open the Web Application. Open a new browser tab, paste the Public IPv4 address, and press Enter. The Address Book application interface loads.
  4. Configure the DB Connection. Select the RDS or Configuration tab in the app. Enter Endpoint = your RDS endpoint, Database = lab, Username = main, Password = lab-password. Click Submit.

Key Concepts Recap

•

Principle of Least Privilege

Security Groups grant only the minimum access required. DBSG allows Port 3306 from one source only — the web tier.

•

Multi-AZ High Availability

By placing the primary and standby in separate AZs, RDS survives a single-zone failure with automatic failover and no data loss.

•

Private Database Placement

Setting Public Access = No and using private subnets ensures the database is never reachable from the internet — only from within the VPC.

•

Managed Service Benefits

Amazon RDS handles OS patching, backups, and replication automatically — freeing you to focus on application logic rather than infrastructure operations.
