1. Building a Serverless API with API Gateway & Lambda

Author a live Node.js function and expose it through a public REST endpoint — no servers involved.

Lab Overview

An Amazon API Gateway REST endpoint acts as the public-facing front door, routing incoming HTTP client traffic directly to a custom Node.js function running inside AWS Lambda — no EC2 instances, no web servers, no infrastructure to maintain. You will author the function, wire it to a POST method, deploy it to a live stage, and validate it end-to-end with a real HTTP request.

The LabRole Rule

Never create custom IAM users, groups, or roles, and never modify root billing settings — doing so triggers an AWS Academy SCP restriction and may lock your learner account. Every resource in this lab must use the pre-provisioned IAM role named LabRole, which already has the permissions this lab needs. Do not create a new role or select any other option from the dropdown.

Step 3: Author the Node.js Application Code

In the Code source panel, double-click index.mjs, delete the existing boilerplate, and paste:

export const handler = async (event) => {
  console.log("Event Received:", JSON.stringify(event, null, 2));

  let clientUserName = "Guest Professional Analyst";
  if (event.body) {
    try {
      const parsedBody = JSON.parse(event.body);
      if (parsedBody.name) { clientUserName = parsedBody.name; }
    } catch (error) {
      console.log("Non-JSON payload. Using defaults.");
    }
  }

  const operationalResponsePayload = {
    status: "SUCCESS_VALIDATION_COMPLETE",
    message: `Hello, ${clientUserName}! Your event executed in AWS Lambda.`,
    timestamp: new Date().toISOString(),
    environment: "AWS_Academy_ITNW1009_Sandbox"
  };

  return {
    statusCode: 200,
    headers: {
      "Content-Type": "application/json",
      "Access-Control-Allow-Origin": "*"
    },
    body: JSON.stringify(operationalResponsePayload)
  };
};

Click the orange Deploy button above the code editor and wait for the confirmation overlay — your function is now live.

Logging & Parsing

console.log writes the full raw event to CloudWatch Logs. The handler then tries to parse a JSON body — if a name key is present, it replaces the default "Guest Professional Analyst" value with the caller's supplied name.

Building the Response

A structured JSON payload is assembled with a status flag, personalized message, timestamp, and environment label, then returned as an HTTP 200 response with CORS support via Access-Control-Allow-Origin: *.

Create Method

Set Integration Type

Enable Proxy Integration

Critical

2. Deploying & Exploring a Managed PaaS Environment with Elastic Beanstalk

Deploy real code to a pre-built environment and see the EC2, load balancer, and Auto Scaling resources running underneath it.

Lab Overview

An AWS Elastic Beanstalk environment has already been created for you. You will deploy application code to it, test that it's live, and inspect the AWS resources Beanstalk provisioned on your behalf — a security group, a load balancer, and an Auto Scaling group — all without configuring any of them by hand. Estimated time: 30 minutes.

Task 1: Access the Elastic Beanstalk Environment

  1. In the console search box, search for and choose Elastic Beanstalk. An Environments page opens listing an existing application (if Health isn't yet Ok, wait a few moments).
  2. Choose the environment name to open its Dashboard — health should show Ok. The environment is ready to host an application, but doesn't yet have running code.
  3. Near the top of the page, choose the Domain link (ends in elasticbeanstalk.com). You should see HTTP Status 404 — Not Found.

This 404 is expected — the application server doesn't have an application running on it yet. Return to the Elastic Beanstalk console for the next step.

Task 2: Deploy a Sample Application

  1. Download the sample Tomcat application from AWS's documentation sample archive (tomcat.zip).
  2. Back in the Elastic Beanstalk Dashboard, choose Upload and Deploy, choose Choose File, and select the downloaded zip.
  3. Choose Deploy. It takes a minute or two for Beanstalk to update the environment.
  4. Once complete, revisit (or refresh) the Domain URL — the deployed web application should now display.
Explore Configuration

Choose Configuration in the left pane. The Instance traffic and scaling panel shows the EC2 security groups, minimum/maximum instance counts, and instance type backing your app. The Networking, database, and tags panel is empty since this environment has no database — opening Edit there shows how easily one could be added, but this activity doesn't require it (choose Cancel to back out).

Explore Monitoring

Under Environment, choose Monitoring and browse the available charts to see what visibility Beanstalk gives you into your running application.

Task 3: Explore the Resources Supporting Your Application

Search for and open EC2, then choose Instances — two instances containing samp in their names are running your web application. Feel free to keep exploring; Beanstalk created these resources on your behalf, but you still have direct access to them:

Security Group

Port 80 open to allow inbound web traffic.

Load Balancer

Both instances are registered as targets behind it.

Auto Scaling Group

Runs 2–6 instances depending on network load.

3. Automating EC2 Shutdown with Lambda & EventBridge

Write a Python automation script and trigger it on a schedule — a real "stopinator" for runaway instances.

Lab Overview

In this activity you will create an AWS Lambda function and an Amazon EventBridge rule that triggers it every minute. The function uses an IAM role that allows it to stop a running EC2 instance in your account — a small but complete example of scheduled, serverless infrastructure automation. Estimated time: 30 minutes.

Task 1: Create a Lambda Function

Search for and open Lambda, choose Create a function, select Author from scratch, and configure:

Function namemyStopinator
RuntimePython 3.11
Execution roleUse an existing role → myStopinatorRole

Expand Change default execution role, select Use an existing role, choose myStopinatorRole from the dropdown, then choose Create function.

Task 2: Configure the EventBridge Trigger

Configuring an EventBridge rule as the function's trigger lets it behave like a cron job or scheduled task — without needing a server to host it.

  1. Choose Add trigger, then select EventBridge (CloudWatch Events) from the trigger dropdown.
  2. For the rule, choose Create a new rule and configure: Rule name everyMinute; Rule type Schedule expression; Schedule expression rate(1 minute).
  3. Choose Add.

A real-world "stopinator" would more likely use a cron expression. A tight 1-minute rate expression is used here purely so you can see results quickly.

Task 3: Configure the Lambda Function Code

Below Function overview, choose Code, open lambda_function.py, delete the existing code, and paste:

import boto3
region = '<REPLACE_WITH_REGION>'
instances = ['<REPLACE_WITH_INSTANCE_ID>']
ec2 = boto3.client('ec2', region_name=region)

def lambda_handler(event, context):
    ec2.stop_instances(InstanceIds=instances)
    print('stopped your instances: ' + str(instances))

After pasting, double-check line 5 for a stray period that sometimes gets added — delete it if present.

  1. Replace <REPLACE_WITH_REGION> with your actual region code (e.g., us-east-1 for N. Virginia — visible in the top-right region dropdown). Keep the single quotes around it.
  2. Challenge: confirm an EC2 instance named instance1 is running, and copy its instance ID (starts with i-) from the EC2 console's Instances list / Details tab.
  3. Back in the Lambda console, replace <REPLACE_WITH_INSTANCE_ID> with that instance ID, keeping the single quotes.
  4. Choose File → Save, then choose Deploy in the Code source box. The function is now fully configured and will attempt to stop your instance every minute.
  5. Choose the Monitor tab to see invocation count, error count, and success-rate charts for your function.

Task 4: Verify That the Lambda Function Worked

  1. Return to the EC2 console browser tab and check whether your instance was stopped (refresh if needed).
  2. Try starting the instance again — within about a minute, the scheduled Lambda invocation will stop it again, since the EventBridge rule keeps firing every minute regardless of who (re)started it.

Day 6 Summary: Key Takeaways

•

Serverless Means No Managed Infrastructure

API Gateway + Lambda deliver a full public API without provisioning a single EC2 instance, load balancer, or OS to patch.

•

Lambda Proxy Integration Is Load-Bearing

Skipping it is the single most common cause of a 502 error — it's what lets the raw HTTP request reach your handler's event object.

•

Beanstalk Is PaaS, Not Magic

Under the hood, Beanstalk is still creating ordinary EC2 instances, a security group, a load balancer, and an Auto Scaling group — it just manages them for you.

•

EventBridge Turns Lambda into a Scheduler

A rate or cron-based EventBridge rule gives you serverless "cron job" behavior — automating tasks like cost-saving EC2 shutdowns with no server required to host the schedule.
