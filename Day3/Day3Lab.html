1. Exploring IAM Users, Groups & Policies

Audit the pre-built Identity and Access Management setup before making any changes.

Lab Overview

AWS Identity and Access Management (IAM) lets you centrally manage users, security credentials, and the permissions that control which AWS resources those users can access. Today you will inspect pre-created IAM users and groups, read the JSON policies attached to each group, then work through a business scenario that assigns users to the correct group and verifies their access is exactly as intended.

Service Restrictions

This sandbox restricts access to only the AWS services and actions needed for this lab. If you try to explore something outside these instructions, expect an Access Denied–style error. "Not authorized" messages that appear while assigning users to groups are expected and will not affect your grade.

Task 1: Accessing the AWS Console & Opening IAM

  1. Choose Start Lab at the top of your lab portal. A countdown timer appears showing your remaining session time — click Start Lab again anytime before it hits 0:00 to extend your session.
  2. Wait until the status circle to the right of the AWS link in the upper-left corner turns solid green, confirming your sandbox is ready.
  3. Click the AWS link to open the AWS Management Console in a new browser tab. Arrange it alongside these instructions so you can follow along without constantly switching windows.
  4. In the search box next to Services, type IAM and select it to open the Identity and Access Management dashboard.

Task 2: Reviewing the Pre-Created Users

  1. In the left navigation pane, choose Users. Three accounts have already been created for you: user-1, user-2, and user-3.
  2. Choose the user-1 link to open its summary page. On the Permissions tab, notice this user has no permissions attached yet.
  3. Choose the Groups tab — user-1 is not a member of any group.
  4. Choose the Security credentials tab. Note that user-1 already has a Console password configured, so it can sign in once it gains permissions.

Task 3: Inspecting the Group Policies

In the left pane, choose User groups. Three groups already exist: EC2-Admin, EC2-Support, and S3-Support. Open each one's Permissions tab and expand its policy with the + icon to see what it allows.

EC2-Support

Attached to the AWS Managed Policy AmazonEC2ReadOnlyAccess. Grants permission to list and describe EC2, Elastic Load Balancing, CloudWatch, and Auto Scaling resources — view only, no changes.

S3-Support

Attached to the AWS Managed Policy AmazonS3ReadOnlyAccess. Grants Get and List actions on S3 buckets and objects — a textbook example of least privilege.

EC2-Admin

Uses an Inline Policy (bound to this one group only, not reusable) that grants Describe access plus the ability to Start and Stop EC2 instances.

{
  "Effect": "Allow",
  "Action": [
    "ec2:Describe*",
    "ec2:StartInstances",
    "ec2:StopInstances"
  ],
  "Resource": "*"
}

Every IAM policy statement follows the same structure: Effect (Allow or Deny), Action (which API calls are covered), and Resource (which AWS resources the rule applies to, with * meaning "any resource").

Business Scenario

Your company is growing its use of AWS and needs to grant new staff access based on their job function. For the rest of this lab, you'll enable exactly this access:

User Target Group Resulting Permissions
user-1 S3-Support Read-only access to Amazon S3
user-2 EC2-Support Read-only access to Amazon EC2
user-3 EC2-Admin View, start, and stop Amazon EC2 instances

2. Assigning Users to Groups & Validating Access

Put each user into their group, then sign in as each one to confirm the permission boundaries actually hold.

Task 4: Add Each User to Their Group

  1. In the left pane, choose User groups and open S3-Support. Choose the Users tab, then click Add users. Select user-1 and click Add users again to confirm.
  2. Return to User groups and repeat the same steps to add user-2 to EC2-Support.
  3. Return to User groups once more and add user-3 to EC2-Admin.
  4. Back on the main User groups view, confirm each group now shows a count of 1 in its Users column. If any group doesn't show 1, revisit the steps above for that user.

Task 5: Retrieve the IAM Sign-In URL

  1. In the left pane, choose Dashboard. On the right, locate the Sign-in URL for IAM users in this account link — it looks similar to https://123456789012.signin.aws.amazon.com/console.
  2. Copy this URL into a text editor; you'll reuse it to sign in as all three test users.
  3. Open a private/incognito browser window (Ctrl+Shift+N in Chrome/Edge, or the equivalent "New Private Window" option in Firefox) so cached credentials from one user don't leak into the next login.

Task 6: Sign In as Each User & Verify Boundaries

Paste the sign-in URL into your private window each time, sign out between users (click the username at the top-right, then Sign Out), and run the checks below.

user-1 · Password: Lab-Password1

S3 — Allowed

Open the S3 console and browse the existing bucket's contents successfully (it will be empty).

EC2 — Blocked

Opening Instances under EC2 returns "You are not authorized to perform this operation."

user-2 · Password: Lab-Password2

EC2 View — Allowed

The LabHost instance is visible (check the region is N. Virginia if it isn't).

Close the private browser window once user-3's test is complete.

Common Lab Troubleshooting Guidance

Observed Symptom Root Cause Fix
"Failed to stop instance: Not authorized" Logged in as user-1 or user-2 — neither has Stop permissions. Expected behavior. Sign out and sign in as user-3 to stop the instance.
"You don't have permissions to list buckets" Logged in as user-2 or user-3 — S3 is outside their policy scope. Expected behavior. Use user-1 to access S3.
Blank instance grid / LabHost not visible The console region has defaulted away from the lab's deployment zone. Open the region dropdown (top-right) and switch to N. Virginia.
Sign-in URL loads blank or errors A browser extension or proxy is blocking the IAM redirect path. Clear cache, use a fresh incognito window, or try a different browser.

Lab Summary: Key Takeaways

•

Permissions Flow Through Groups

Users inherit permissions by joining a group — they aren't granted permissions directly. Groups are how enterprise access control scales.

•

Managed vs. Inline Policies

Managed policies are reusable and centrally maintained; inline policies are custom and bound to a single user or group for one-off needs.

•

Principle of Least Privilege

Each user could only do what their policy explicitly allowed — every attempt to exceed that was blocked immediately, exactly as designed.

•

Test in Isolation

Signing in as each user from a private/incognito window keeps credentials from bleeding between sessions — a real-world habit for validating IAM policy behavior.
