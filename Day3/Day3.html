Identity & Access Analogy Library

Breaking down Shared Responsibility, IAM, and least-privilege access concepts for learners with no prior IT background.

Metaphor A

The Apartment Building

A large apartment complex has a landlord who maintains the building's structure, wiring, plumbing, and front-gate security system. Each tenant is responsible for locking their own apartment door, deciding who gets a copy of their key, and reporting any suspicious activity inside their own unit. Neither party can do the other's job for them.

Abstract Concept Map

  • Shared Responsibility: The landlord (AWS) secures the building's foundation and shared systems; the tenant (customer) secures everything inside their own apartment.
  • IAM User: One named tenant with their own personal key to their own apartment.
  • IAM Group: All tenants on the same floor sharing the same floor-access privileges, rather than each having a uniquely negotiated deal.
  • IAM Role: A temporary valet key handed to a repair technician for the afternoon, which stops working once the job is done.
  • Policy Document (JSON): The written lease agreement that spells out exactly which doors, hallways, and amenities a tenant's key is allowed to open.
  • MFA: Requiring both a key card swipe and a PIN code before the lobby door will unlock.
  • Least Privilege: Giving the cleaning crew a key that only opens hallway closets, not a master key to every unit in the building.
  • Principal: Whoever is actually holding the key and requesting entry — a tenant, a guest, or a maintenance worker.
Metaphor B

The Corporate Badge Access System

A large office tower has a private security company that guards the building's main entrances, elevators, and fire systems. Once inside, it is every department's own job to decide which employee badges can open which internal doors, and to lock up their own file cabinets and server closets at night.

Abstract Concept Map

  • Shared Responsibility: The building's security company (AWS) protects the shared perimeter and infrastructure; each department (the customer) is responsible for locking its own internal doors and cabinets.
  • IAM User: One employee's personal photo ID badge, issued to them specifically.
  • IAM Group: The "Accounting Department" badge tier, where every accountant automatically gets the same set of door permissions.
  • IAM Role: A one-day visitor pass issued to an outside contractor, which expires automatically at the end of their visit.
  • Policy Document (JSON): The access-control list on file with security stating precisely which doors a given badge is permitted to open.
  • MFA: Requiring a badge swipe plus a fingerprint scan before the server room door will open.
  • Least Privilege: Giving a summer intern a badge that opens the break room and their own desk area, but not the executive floor or the server room.
  • Principal: Whoever is presenting the badge and requesting access — an employee, a contractor, or even an automated delivery robot with its own access credential.

Vocational Case Study: The Leaked Admin Keys

An incident analysis of a developer who accidentally published admin API access keys on a public GitHub repository.

The Weekend Cryptomining Bill

Marcus, a junior developer at a small logistics startup, was rushing to finish a personal side project on a Friday night. To save time, he hardcoded his company AWS account's long-term admin access key and secret key directly into a Python script, then pushed the whole project — including that script — to a brand-new public GitHub repository so he could show it to a friend. He meant to make the repository private later that weekend, but forgot.

Automated bots constantly scan public GitHub repositories for exposed credential patterns, and within roughly twenty minutes of the push, Marcus's admin access key had already been scraped and tested by an attacker. Because the leaked key belonged to a long-lived IAM User with full administrator permissions rather than a narrowly scoped Role, the attacker was able to spin up dozens of large EC2 instances across multiple Regions to mine cryptocurrency. By the time the company's finance team noticed an alarming spike in their AWS bill on Monday morning, the unauthorized charges had already climbed into the thousands of dollars.

The company's cloud administrator, Priya, immediately deactivated the leaked access key, deleted the rogue EC2 instances, and enabled AWS Budgets alerts and MFA enforcement on every IAM User going forward. In the post-incident review, the team concluded that the real root cause was not the GitHub mistake alone, but a deeper policy gap: developers had been issued permanent admin-level access keys instead of temporary, least-privilege IAM Roles scoped to only what each project actually required.

Interactive Analysis Checkpoint

Correct answers: 1-A, 2-A, 3-B.

Evaluation Criteria for Grading Essay/Oral Answers:

  • Excellent (5 pts): Correctly identifies the long-term admin key (not the GitHub mistake alone) as the root structural flaw, explains how Least Privilege and short-lived Roles limit blast radius, and correctly places credential management under the customer's Security IN the cloud responsibility.
  • Developing (3 pts): Identifies that leaking a key is bad, but is fuzzy on why an admin-level long-term key is worse than a scoped Role, or is unsure who is responsible for the mistake under the Shared Responsibility Model.
  • Unsatisfactory (0-1 pts): Blames GitHub or AWS entirely and cannot explain any preventative IAM control.
