Lab 1, Part 1: Guided Lab — Build Your VPC & Launch a Web Server

Follow the pre-packaged Curriculum track: build a VPC, attach an Internet Gateway, and deploy a live web server.

Two Learning Tracks

This lab covers two distinct environments. The Guided Lab (this tab) is a pre-packaged, step-by-step curriculum environment with automatic grading checkpoints — ideal for a structured first pass. The Sandbox Lab (next tab) is unguided, from-scratch architecture design within a $100 budget, for learners ready to engineer a secure two-tier VPC independently.

Critical Curriculum Constraints

Never create custom IAM Users, Groups, or root organizational policy changes — the restricted Academy Student profile blocks these and generates an Access Denied error that breaks your lab score tracker. Always attach the pre-configured LabRole IAM instance profile when launching instances or authorizing infrastructure permissions. Missing this step causes instance validation to fail.

Task 1: Access the Lab Sandbox

  1. Go to the Modules tab and select Lab 2.
  2. Click Start Lab, then wait until the status icon turns from blinking amber to solid green.
  3. Click the green AWS link to open the Management Console. Clicking before the status is green may result in a partial or failed session.
  4. Work inside a private/incognito browser window to prevent cached credentials from interfering with the AWS Console authentication flow.

Lab 1, Part 2: Sandbox Lab — Engineer a Two-Tier Network from Scratch

Design and deploy an isolated two-tier VPC with a NAT Gateway, least-privilege security groups, and a private-only test instance.

Sandbox Cost Controls & Boundaries

The Sandbox Lab allocates a $100 budget cap per student. Stick strictly to services specified in the lab, always select t2.micro or t3.micro instance types, and execute the full teardown sequence immediately after validation — NAT Gateways and Elastic IPs continue billing even when idle.

Task 1: Initialize the Sandbox Environment

  1. Log in to your AWS Academy portal home screen and navigate to the Learner Lab section.
  2. Click Start Lab and wait until the status circle turns solid green.
  3. Click the AWS Console link to launch your sandbox workspace — a clean, empty AWS account with no pre-configured resources.
  4. Keep a plain text editor open to record dynamically generated resource IDs, Elastic IPs, and gateway identifiers as you build.

Task 2: Build a Two-Tier VPC from Scratch

Use the VPC and more wizard, which automatically creates subnets, route tables, and an Internet Gateway in one guided flow:

Expand the Subnet CIDR blocks panel to confirm the wizard assigned: Public subnet 192.168.0.0/24, Private subnet 192.168.1.0/24. NAT is set to None here deliberately — you'll configure it manually in Task 3 to control costs and understand the provisioning process.

Task 3: Configure an Outbound-Only NAT Gateway

A NAT Gateway lets private-subnet instances initiate outbound connections (e.g., software updates) while blocking inbound connections from external sources.

Critical Placement

Place the NAT Gateway inside the Public Subnet, not the Private Subnet. Placing it in the wrong subnet breaks outbound routing for your private tier.

  1. Confirm Sandbox-Net-igw was auto-created and attached to your VPC by the wizard.
  2. Create a NAT Gateway named Sandbox-NAT-Gateway, Subnet: Sandbox-Net-subnet-public, Connectivity type: Public.
  3. Click Allocate Elastic IP to assign a static public IP, then click Create NAT gateway. Allow up to 2 minutes for provisioning.

Task 4: Engineering Private Subnet Routing

  1. Navigate to Route tables and create Private-Route-Table.
  2. Add a route for destination 0.0.0.0/0 targeting your NAT Gateway, and save changes.
  3. Switch to the Subnet associations tab, check only Sandbox-Net-subnet-private, and save associations.

Task 5: Implement Stateful Instance-Level Firewalls

Create a least-privilege security group that allows only your public-subnet application tier to reach database ports, blocking all external IPs:

Security Group Settings
  • Name: App-DB-SG
  • Description: Restricted database port filter
  • VPC: Sandbox-Net
Inbound Rule
  • Type: MySQL/Aurora (3306) or MS SQL (1433) — match your DB variant
  • Source type: Custom
  • Source CIDR: 192.168.0.0/24 (Public Subnet only)

Structural Validation: Launch a Private Test Instance

A correctly isolated instance should have a private IP address only, with no public IP assigned.

  1. Go to the EC2 dashboard, click Launch instances, name it SandboxStorageHost, and select an approved free-tier t2.micro AMI.
  2. Under Network settings, select Sandbox-Net as the VPC, choose the Private Subnet, and set Auto-assign public IP to Disable.
  3. Attach the existing App-DB-SG security group, and in Advanced details attach the LabRole IAM instance profile, then click Launch instance.
  4. In the instance details pane, confirm a private IP in the 192.168.1.x range with no public IP — this confirms your network isolation is working correctly.

Lab 2, Part 1: Build Your VPC

Use the "VPC and more" wizard to stand up a multi-AZ network with public and private subnets, route tables, and a NAT Gateway.

Lab Overview

Amazon VPC lets you launch AWS resources into a virtual network you define — closely resembling a traditional data center network, with the scalability of AWS. This lab creates a VPC spanning two Availability Zones, then configures a security group and launches an EC2 web server into it. Estimated time: 30 minutes.

Task 1: Create Your VPC with "VPC and More"

Search for and open VPC, confirm the region reads N. Virginia (us-east-1), choose the VPC dashboard link, then choose Create VPC (or Launch VPC Wizard if that's what appears). Select VPC and more and configure:

Core Settings
  • Name tag auto-generation: keep Auto-generate, change value from "project" to lab
  • IPv4 CIDR block: keep 10.0.0.0/16
  • Number of Availability Zones: 1
  • Number of public subnets: keep 1; private subnets: keep 1
Subnet CIDRs & NAT
  • Expand Customize subnets CIDR blocks
  • Public subnet CIDR (us-east-1a): 10.0.0.0/24
  • Private subnet CIDR (us-east-1a): 10.0.1.0/24
  • NAT gateways: In 1 AZ; VPC endpoints: None
  • Keep DNS hostnames and DNS resolution both enabled

Confirm the Preview panel shows VPC lab-vpc, subnets lab-subnet-public1-us-east-1a / lab-subnet-private1-us-east-1a, route tables lab-rtb-public / lab-rtb-private1-us-east-1a, and network connections lab-igw / lab-nat-public1-us-east-1a. Choose Create VPC and wait for all resources — including the NAT Gateway — to finish provisioning before choosing View VPC.

Internet Gateway

Allows communication between EC2 instances in your VPC and the internet. The public subnet's route table sends 0.0.0.0/0 traffic here — that's what makes it "public."

NAT Gateway

Provides internet connectivity to instances in private subnets without giving them a direct connection to the internet gateway.

Task 2: Create Additional Subnets in a Second AZ

Subnets in multiple Availability Zones support High Availability designs. In Subnets, choose Create subnet twice:

Second Public Subnet
  • VPC ID: lab-vpc
  • Subnet name: lab-subnet-public2
  • Availability Zone: second AZ (e.g., us-east-1b)
  • IPv4 CIDR: 10.0.2.0/24
Second Private Subnet
  • VPC ID: lab-vpc
  • Subnet name: lab-subnet-private2
  • Availability Zone: second AZ (e.g., us-east-1b)
  • IPv4 CIDR: 10.0.3.0/24

Next, associate route tables with the new subnets. In Route tables, select lab-rtb-private1-us-east-1a → Routes tab — note Destination 0.0.0.0/0 targets the NAT Gateway. Choose Subnet associations → Edit subnet associations, leave lab-subnet-private1-us-east-1a selected, also select lab-subnet-private2, then Save associations.

Repeat for the public side: select lab-rtb-public → confirm its Routes tab shows Destination 0.0.0.0/0 targeting the Internet Gateway → Subnet associations → Edit → keep lab-subnet-public1-us-east-1a selected, also select lab-subnet-public2, then Save associations.

Your VPC now has public and private subnets configured across two Availability Zones, with both route tables properly updated.

Lab 2, Part 2: Security Group & Web Server Launch

Open HTTP access, then launch a fully bootstrapped PHP web application into your new VPC.

Task 3: Create a VPC Security Group

A security group acts as a virtual firewall for any instance it's attached to. In Security groups, choose Create security group:

Basic Configuration
  • Security group name: Web Security Group
  • Description: Enable HTTP access
  • VPC: remove the default, select lab-vpc
Inbound Rule
  • Type: HTTP
  • Source: Anywhere-IPv4
  • Description: Permit web requests

Scroll down and choose Create security group. You'll attach it to your EC2 instance next.

Task 4: Launch a Web Server Instance

Search for and open EC2, then choose Launch instance:

Identity & Compute
  • Name: Web Server 1
  • AMI: keep default Amazon Linux → Amazon Linux 2023
  • Instance type: keep default t2.micro
  • Key pair: select vockey
Network Settings
  • Network: lab-vpc
  • Subnet: lab-subnet-public2 (not Private!)
  • Auto-assign public IP: Enable
  • Firewall: select existing → Web Security Group

Keep the default storage settings (8 GiB gp3 root volume). Expand Advanced details, scroll to User data, and paste:

#!/bin/bash
# Install Apache Web Server and PHP
dnf install -y httpd wget php mariadb105-server
# Download Lab files
wget https://aws-tc-largeobjects.s3.us-west-2.amazonaws.com/CUR-TF-100-ACCLFO-2/2-lab2-vpc/s3/lab-app.zip
unzip lab-app.zip -d /var/www/html/
# Turn on web server
chkconfig httpd on
service httpd start

This script runs once at first boot with root permissions — it installs Apache, PHP, and MariaDB libraries, then downloads and installs a sample PHP web application.

Choose Launch instance, then View all instances. Wait until Web Server 1 shows 2/2 checks passed (refresh every ~30 seconds).

Verify the Web Server

  1. Select Web Server 1 and copy the Public IPv4 DNS value from the Details tab.
  2. Open a new browser tab, paste the Public DNS value, and press Enter.
  3. You should see a web page displaying the AWS logo and instance metadata values — confirming your VPC, security group, and bootstrap script all worked together correctly.

Day 8 Summary: Key Takeaways

•

Public vs. Private Is Just Routing

What makes a subnet "public" or "private" is entirely defined by its route table — pointing to an Internet Gateway versus a NAT Gateway.

•

NAT Gateways Must Sit in the Public Subnet

A NAT Gateway physically needs internet reachability itself to forward traffic on behalf of private instances — placing it in the private subnet breaks the whole chain.

•

Least Privilege Applies at the Port Level

Restricting a database security group's source CIDR to just the application subnet — instead of 0.0.0.0/0 — blocks all external scanning attempts by default.

•

LabRole Is Non-Negotiable

Every instance and permission-bearing action in the Guided track depends on the pre-provisioned LabRole — skipping it is the most common cause of a failed validation checkpoint.
