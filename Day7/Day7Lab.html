Lab 1, Part A: Working with Amazon Elastic Block Store (EBS)

Provision, attach, format, mount, and snapshot a persistent block storage volume on a live EC2 instance.

Lab Overview

Amazon EBS provides persistent, high-performance raw block storage that behaves exactly like a physical internal hard disk attached to your server. In this 75-minute guided lab, you'll gain career-ready skills in provisioning, formatting, mounting, and backing up EBS volumes on a running compute instance.

Critical Permission Rules

AWS Academy environments are governed by Service Control Policies (SCPs). Never create custom IAM Users, Groups, or root billing notifications. Always use the pre-provisioned security profile named LabRole when authorizing instances or writing policies. Violating these triggers immediate "Access Denied" errors.

Task 1: Initialize the Pre-Packaged Environment

  1. Log into your AWS Academy Learning Portal (Canvas) and locate Lab 4: Working with EBS in the navigation links.
  2. Click Start Lab. A countdown timer and status circle appear — wait until the circle turns from amber to solid green.
  3. Click the AWS text link directly to the left of the green status light to open an authenticated Management Console session.

Task 2: Create a Standalone EBS Volume

Navigate to EC2 → Elastic Block Store → Volumes using the top search bar. Click Create volume and configure:

SettingRequired ValueWhy It Matters
Volume typeGeneral Purpose SSD (gp3)Default; within Academy sandbox cost limits
Size10 GiB exactlyLarger sizes are blocked by SCPs
Availability ZoneMatch your running EC2 instanceEBS volumes cannot attach across AZs
AZ Alignment Is Critical

An EBS volume works like an internal hard disk over a low-latency local bus — it is structurally impossible to attach to an EC2 instance in a different Availability Zone. Always verify your AZ before clicking Create volume.

Task 3: Attach the Volume to LabHost

  1. In the Volumes table, check the box next to your new 10 GiB volume and confirm its state reads Available.
  2. Click Actions → Attach volume. In the Instance field, select the pre-provisioned virtual machine named LabHost.
  3. Leave the default device path (e.g., /dev/sdf or /dev/xvdf) unchanged and click Attach volume.
  4. Watch the status column refresh from Available to In-use, confirming successful attachment.

Task 4: Connect via CLI & Mount the Disk

Go to EC2 → Instances, select LabHost, click Connect, choose the EC2 Instance Connect tab (username ec2-user), and click Connect to open a browser-based Linux terminal. Run each command in sequence:

Step 1 — Verify Disk Attachment
lsblk

You should see a secondary device (xvdf, ~10G) alongside the root disk.

Step 2 — Format with XFS
sudo mkfs -t xfs /dev/xvdf

Applies an XFS file system to the raw disk. Do not skip — mounting unformatted storage will fail.

Steps 3–4 — Mount Point & Mount
sudo mkdir /mnt/data-store
sudo mount /dev/xvdf /mnt/data-store

Creates the directory and binds the formatted disk to it.

Step 5 — Verify & Write Test File
df -h
echo "ITNW 1009 EBS Lab Verification Complete" | sudo tee /mnt/data-store/lab-check.txt

The df -h output must show /dev/xvdf mounted at /mnt/data-store with ~10G capacity.

Task 5: Create an EBS Snapshot Backup

Snapshots are point-in-time backups stored in Amazon S3, essential for enterprise data durability. Without closing your terminal tab, navigate to Snapshots and choose Create Snapshot. In the Description field, enter exactly:

EBS Automated Class Backup Snapshot Target

Monitor status until it changes from Pending to Completed, then return to the Academy dashboard and click Submit / Check Progress to lock in your grade points.

Lab 1, Part B: Sandbox Scratch Build — VPC, EC2, S3 & Glacier Deep Archive

Act as Lead Cloud Engineer: build the network, deploy a host, and automate long-term archival — all from scratch.

Lab Overview

In Part B you step into the role of a Lead Cloud Systems Engineer, building a complete infrastructure environment from absolute scratch — no templates, no pre-made resources — inside the unguided AWS Academy Learner Lab Sandbox.

$100 Budget — Hard Limit

Never leave unused Elastic IP allocations running, do not provision oversized EBS volumes, and always terminate resources at the end of every session. Key Concepts: Storage Services Compared

Amazon EBS

Persistent block storage attached to a single EC2 instance. Behaves like a physical internal hard disk. Must be in the same AZ as its instance. Ideal for OS volumes, databases, and low-latency I/O.

Amazon S3

Scalable object storage accessible from anywhere. Stores files inside globally-named buckets. Ideal for backups, static websites, data lakes, and media delivery.

S3 Glacier Deep Archive

The lowest-cost AWS storage class for long-term compliance archiving. Retrieval takes up to 12 hours. Best for regulatory records and audit logs retained for 7–10+ years.

Lab 2, Part 1: Build Your VPC

Use the "VPC and more" wizard to stand up a multi-AZ network with public and private subnets, route tables, and a NAT Gateway.

Lab Overview

Amazon VPC lets you launch AWS resources into a virtual network you define — closely resembling a traditional data center network, with the scalability of AWS. This lab creates a VPC spanning two Availability Zones, then configures a security group and launches an EC2 web server into it. Estimated time: 30 minutes.

Task 1: Create Your VPC with "VPC and More"

Search for and open VPC, confirm the region reads N. Virginia (us-east-1), choose the VPC dashboard link, then choose Create VPC (or Launch VPC Wizard if that's what appears). Select VPC and more and configure:

Core Settings
  • Name tag auto-generation: keep Auto-generate, change value from "project" to lab
  • IPv4 CIDR block: keep 10.0.0.0/16
  • Number of Availability Zones: 1
  • Number of public subnets: keep 1; private subnets: keep 1
Subnet CIDRs & NAT
  • Expand Customize subnets CIDR blocks
  • Public subnet CIDR (us-east-1a): 10.0.0.0/24
  • Private subnet CIDR (us-east-1a): 10.0.1.0/24
  • NAT gateways: In 1 AZ; VPC endpoints: None
  • Keep DNS hostnames and DNS resolution both enabled

Confirm the Preview panel shows VPC lab-vpc, subnets lab-subnet-public1-us-east-1a / lab-subnet-private1-us-east-1a, route tables lab-rtb-public / lab-rtb-private1-us-east-1a, and network connections lab-igw / lab-nat-public1-us-east-1a. Choose Create VPC and wait for all resources — including the NAT Gateway — to finish provisioning before choosing View VPC.

Internet Gateway

Allows communication between EC2 instances in your VPC and the internet. The public subnet's route table sends 0.0.0.0/0 traffic here — that's what makes it "public."

NAT Gateway

Provides internet connectivity to instances in private subnets without giving them a direct connection to the internet gateway.

Task 2: Create Additional Subnets in a Second AZ

Subnets in multiple Availability Zones support High Availability designs. In Subnets, choose Create subnet twice:

Second Public Subnet
  • VPC ID: lab-vpc
  • Subnet name: lab-subnet-public2
  • Availability Zone: second AZ (e.g., us-east-1b)
  • IPv4 CIDR: 10.0.2.0/24
Second Private Subnet
  • VPC ID: lab-vpc
  • Subnet name: lab-subnet-private2
  • Availability Zone: second AZ (e.g., us-east-1b)
  • IPv4 CIDR: 10.0.3.0/24

Next, associate route tables with the new subnets. In Route tables, select lab-rtb-private1-us-east-1a → Routes tab — note Destination 0.0.0.0/0 targets the NAT Gateway. Choose Subnet associations → Edit subnet associations, leave lab-subnet-private1-us-east-1a selected, also select lab-subnet-private2, then Save associations.

Repeat for the public side: select lab-rtb-public → confirm its Routes tab shows Destination 0.0.0.0/0 targeting the Internet Gateway → Subnet associations → Edit → keep lab-subnet-public1-us-east-1a selected, also select lab-subnet-public2, then Save associations.

Your VPC now has public and private subnets configured across two Availability Zones, with both route tables properly updated.

Task 4: Launch a Web Server Instance

Search for and open EC2, then choose Launch instance:

Identity & Compute
  • Name: Web Server 1
  • AMI: keep default Amazon Linux → Amazon Linux 2023
  • Instance type: keep default t2.micro
  • Key pair: select vockey
Network Settings
  • Network: lab-vpc
  • Subnet: lab-subnet-public2 (not Private!)
  • Auto-assign public IP: Enable
  • Firewall: select existing → Web Security Group

Keep the default storage settings (8 GiB gp3 root volume). Expand Advanced details, scroll to User data, and paste:

#!/bin/bash
# Install Apache Web Server and PHP
dnf install -y httpd wget php mariadb105-server
# Download Lab files
wget https://aws-tc-largeobjects.s3.us-west-2.amazonaws.com/CUR-TF-100-ACCLFO-2/2-lab2-vpc/s3/lab-app.zip
unzip lab-app.zip -d /var/www/html/
# Turn on web server
chkconfig httpd on
service httpd start

This script runs once at first boot with root permissions — it installs Apache, PHP, and MariaDB libraries, then downloads and installs a sample PHP web application.

Choose Launch instance, then View all instances. Wait until Web Server 1 shows 2/2 checks passed (refresh every ~30 seconds).

Verify the Web Server

  1. Select Web Server 1 and copy the Public IPv4 DNS value from the Details tab.
  2. Open a new browser tab, paste the Public DNS value, and press Enter.
  3. You should see a web page displaying the AWS logo and instance metadata values — confirming your VPC, security group, and bootstrap script all worked together correctly.

Day 7 Summary: Key Takeaways

•

EBS Is AZ-Locked

A volume can only attach to an instance in the exact same Availability Zone — there's no cross-AZ attachment path.

•

Format Before You Mount

A raw block device must be formatted with a file system (like XFS) before it can be mounted — skipping this is the most common lab failure.

•

Lifecycle Policies Automate Cost Savings

S3 lifecycle rules can silently migrate aging data into Glacier Deep Archive, cutting storage costs with zero ongoing manual effort.

•

Public vs. Private Subnets Are Just Routing

What makes a subnet "public" or "private" is entirely defined by its route table — pointing to an Internet Gateway versus a NAT Gateway.
